Dockerman 的第一阶段 Kubernetes 支持提供了一种轻量级、以 GUI 为先的方式，让你无需切换工具或学习新仪表板即可运行和检查真实的 Kubernetes 集群。

## 为什么使用 Dockerman 管理 Kubernetes

当你希望在管理 Docker 的同一应用中使用 Kubernetes 时，请选择 Dockerman。保持界面本地化且快速，随身携带 k3d 实现一键集群设置，并始终停留在原生 Rust 和 Tauri 桌面应用中，无需跳转到其他仪表板。它比 Docker Desktop 内置的 Kubernetes 路径更轻量，适用于本地开发、小型集群和 CI 健康检查。

## 仪表板

打开 Kubernetes 概览查看集群的实时摘要：节点数量和状态、运行中和待定的 Pod、Deployment、Service 以及最近的集群事件。在深入查看具体资源之前，可以将仪表板作为起点。

## 两种启动方式

<Tabs items={['k3d（本地）', '现有集群']}>
  <Tab>
    当你需要用于开发或测试的一次性本地集群时，打开[集群](/zh/docs/kubernetes/cluster)并使用[使用 k3d 启动本地集群](/zh/docs/kubernetes/cluster#使用-k3d-启动本地集群)流程。
  </Tab>
  <Tab>
    当你已有 kind、minikube、EKS、GKE、AKS 或本地部署的 kubeconfig 时，打开[集群](/zh/docs/kubernetes/cluster)并使用[导入已有集群](/zh/docs/kubernetes/cluster#导入已有集群)流程。
  </Tab>
</Tabs>

## 命名空间

使用顶部栏中的命名空间切换器来限定每个 Kubernetes 列表的范围。无需断开集群连接即可切换命名空间，然后继续在新命名空间中浏览工作负载、网络、配置和 RBAC。

## 暂不支持的功能

<Callout type="info">
  第一阶段专注于上述资源。你暂时还无法使用以下功能：

  - k3d 之外的集群创建，如 kubeadm 或托管服务配置。
  - Horizontal Pod Autoscaler。
  - Ingress 之外的 Gateway API。
  - 可视化 CRD 编辑器，请改用 YAML 编辑。
</Callout>
